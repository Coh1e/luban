# `luban new`

Scaffold a new C++ project from a luban template, with `luban.cmake` already
wired up and (by default) an initial build done so clangd works the moment
you open the directory in your editor.

## Synopsis

```text
luban new {app|lib} <name> [--at <dir>] [--no-build] [--target {native|wasm}] [--with-specs]
```

## Project types

| Kind | Template | What it produces | When to use |
|---|---|---|---|
| `app` | `templates/app/` (default) | One executable target at `src/<name>/main.cpp` | A program; CLIs, services, games. |
| `lib` | `templates/app/` (today) | Same scaffold; the actual library/exe split is done via `luban target add lib mylib` later | Library projects you intend to grow with multiple targets. |

> **Note**: as of v0.2 the `app` and `lib` kinds share the same scaffold. The
> `lib` keyword is reserved so the CLI shape can grow without breaking
> existing scripts. For a real "library only" project today, run
> `luban new app foo` then `luban target remove foo` + `luban target add lib mylib`.

## Auto-build behavior

`luban new` runs `luban build` once at the end (skip with `--no-build`).
This:

- Configures cmake with the auto-selected preset (`default` for native,
  `wasm` for `--target=wasm`).
- Runs ninja.
- Copies `build/<preset>/compile_commands.json` to the project root so
  clangd / VSCode pick it up immediately.

Why eager-build by default? Because clangd attached to a project without
a `compile_commands.json` is silently broken (no diagnostics, no autocomplete).
Better to spend 5 seconds at scaffold time than have the user open
`main.cpp` and wonder why nothing works.

The build can fail (e.g., vcpkg not yet bootstrapped on this machine
when the project starts depending on packages). In that case `luban new`
still scaffolds the directory tree successfully вЂ” only the build step
fails and gets reported. Run `luban setup` then `luban build` to recover.

## Flags

| Flag | Default | Effect |
|---|---|---|
| `--at <dir>` | cwd | Parent directory for the new project. The `<name>/` dir is created under here. |
| `--no-build` | off | Skip the initial build step. Faster scaffold; clangd will be unhappy until you `luban build` manually. |
| `--target {native\|wasm}` | `native` | Pick the template tree. `wasm` requires `luban setup --with emscripten` to have been run already. |
| `--with-specs` | off | After scaffold, run `luban specs init` to drop `AGENTS.md` + `specs/` for AI agent integration. |

## Template substitution

`{{name}}` placeholders are expanded in:

- File **contents** вЂ” e.g., `vcpkg.json`'s `"name": "{{name}}"`, `main.cpp`'s
  `std::println("hello from {{name}}!")`.
- File **path components** вЂ” `templates/app/src/{{name}}/main.cpp` becomes
  `<project>/src/<actual-name>/main.cpp`.

Other supported substitutions (sniff the templates for the full set):

- `{{cpp_std}}` вЂ” C++ standard from `luban.toml`, defaults to `23`.
- `{{vcpkg_baseline}}` вЂ” vcpkg-configuration.json baseline commit hash.

## Name validation

Project names must match `^[a-z][a-z0-9_-]*$` вЂ” lowercase, digits, `-` /
`_`, starts with a letter. Why so strict?

- cmake's `project(<name>)` is sensitive to weird characters.
- The name doubles as a target name (`add_executable(<name> ...)`) which
  flows into binary file names.
- The name appears in `vcpkg.json` which is consumed by vcpkg's parser.
- Cross-platform path safety (no spaces, no Unicode in v1).

## After `luban new`

```bat
cd hello
nvim src/hello/main.cpp     # or VS Code, or any clangd-enabled editor
```

Add a vcpkg dependency:

```bat
luban add fmt              # edits vcpkg.json + regenerates luban.cmake
luban build                # vcpkg auto-fetches fmt; rebuilds; updates compile_commands.json
```

Add another target (separate exe / static lib in the same project):

```bat
luban target add exe second-tool       # creates src/second-tool/{CMakeLists.txt,main.cpp}
luban target add lib mycore            # creates src/mycore/{CMakeLists.txt,*.cpp,*.h}
```

## Common pitfalls

- **`--target=wasm` without emscripten**: the scaffold succeeds, but
  `luban build` fails when `emcmake` isn't on PATH. Fix:
  `luban setup --with emscripten`. See [the WASM workflow](../workflows/wasm.md).
- **Directory already exists**: `luban new` refuses to write into a
  pre-populated dir. Either pick a different name or `rm -rf` first.
- **Name collision with vcpkg port**: e.g., `luban new app fmt` вЂ” the
  project's vcpkg.json `"name": "fmt"` then conflicts with the fmt port
  if you `luban add fmt`. Avoid using vcpkg port names as project names.
- **Network needed at scaffold**: `--no-build` skips the network
  dependency, but the initial template copy is local-only either way.
