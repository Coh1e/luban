# `luban new`

Scaffold a new project (or blueprint file) from a luban template.
For project kinds, `luban.cmake` is wired up and (by default) an initial build
is run so clangd works the moment you open the directory.

## Synopsis

```text
luban new <kind> <name> [--at <dir>] [--no-build]

# <kind> = one of:
#   app        executable C++ project
#   lib        library C++ project (with install + export rules)
#   wasm-app   executable C++ project targeting wasm via emscripten
#   blueprint  user blueprint file (not a project)
```

## Project kinds

| Kind | What it produces | Distribution |
|---|---|---|
| `app` | 1 executable target at `src/<name>/main.cpp` | Ship the exe / zip / GitHub release |
| `lib` | 1 library target with `install + export` rules | Package as a vcpkg Port → registry; consumers `luban add` |
| `wasm-app` | 1 executable compiled to wasm via emscripten | `index.html` + `.wasm` |
| `blueprint` | A `~/.config/luban/blueprints/<name>.lua` file (NOT a project) | Apply via `luban bp apply <name>` |

The kind is recorded in `luban.toml [project] kind = "..."` so that `luban build`,
`doctor`, and (future) `luban port new --from-project` can all branch on it.

## What you get

For `luban new app hello`:

```
hello/
├── CMakeLists.txt              # 4 lines, user-owned. Includes luban.cmake.
├── luban.cmake                 # luban-managed; GIT-TRACKED.
│                               # Auto-regenerated by `luban add` / `target add`.
├── luban.toml                  # [project] kind = "app", cpp = 23, ...
├── vcpkg.json                  # {"name":"hello","version":"0.1.0","dependencies":[]}
├── vcpkg-configuration.json    # baseline pinned to a specific vcpkg commit
├── CMakePresets.json           # Ninja generator + vcpkg toolchain via $env{VCPKG_ROOT}
├── compile_commands.json       # generated by initial build (clangd needs this)
├── .gitignore                  # build/, vcpkg_installed/, .cache/, .vs/
├── .clang-format
├── .clang-tidy
├── .vscode/
│   ├── settings.json
│   └── extensions.json
└── src/
    └── hello/
        ├── CMakeLists.txt      # 2 lines: add_executable + luban_apply
        └── main.cpp            # std::println("hello from hello!");
```

For `luban new lib mylib`, the scaffold additionally includes proper
`install(TARGETS ... EXPORT ...)` / `install(EXPORT ...)` /
`CMakePackageConfigHelpers` boilerplate so the lib is **immediately
installable** and ready to be packaged as a vcpkg port.

For `luban new wasm-app foo`, the preset adds a `wasm` preset that uses
`emcmake`; main.cpp shows an `EM_ASM` example; output includes `shell.html`.

For `luban new blueprint dev`, the only output is
`~/.config/luban/blueprints/dev.lua` — a Lua DSL skeleton with a `tools`
table, a `configs` table, and a `meta.requires = { "embedded:cpp-base" }`
hint.

## Auto-build behavior (project kinds only)

`luban new` runs `luban build` once at the end (skip with `--no-build`).
This:
- Configures cmake with the auto-selected preset based on `kind`
  (`default` for app/lib, `wasm` for wasm-app).
- Runs ninja.
- Copies `build/<preset>/compile_commands.json` to the project root so
  clangd / VSCode pick it up immediately.

Why eager-build by default? clangd attached to a project without a
`compile_commands.json` is silently broken (no diagnostics, no autocomplete).
Better to spend 5 seconds at scaffold time than have the user open
`main.cpp` and wonder why nothing works.

## Flags

| Flag | Default | Effect |
|---|---|---|
| `--at <dir>` | cwd | Parent directory for the new project (project kinds only). |
| `--no-build` | off | Skip the initial build step (project kinds only). |

## Template substitution

`{{name}}` placeholders expand in:
- File **contents** — e.g., `vcpkg.json`'s `"name": "{{name}}"`, `main.cpp`'s `std::println("hello from {{name}}!")`.
- File **path components** — `templates/app/src/{{name}}/main.cpp` → `<project>/src/<actual-name>/main.cpp`.

Other supported substitutions:
- `{{cpp_std}}` — C++ standard from `luban.toml`, defaults to `23`.
- `{{vcpkg_baseline}}` — vcpkg-configuration.json baseline commit hash.
- `{{kind}}` — propagated to `luban.toml [project] kind`.

## Name validation

Project names must match `^[a-z][a-z0-9_-]*$` — lowercase, digits, `-` / `_`,
starts with a letter:
- cmake's `project(<name>)` is sensitive to weird characters
- The name doubles as a default target name and binary file name
- The name appears in `vcpkg.json` which is consumed by vcpkg's parser
- Cross-platform path safety (no spaces, no Unicode)

## After `luban new`

For project kinds:

```bat
cd hello
luban add fmt                     :: edits vcpkg.json + regenerates luban.cmake
luban build                       :: vcpkg auto-fetches fmt; rebuilds
luban target add hello-core --lib :: add internal lib (creates src/hello-core/...)
```

For blueprint kind:

```bat
vim ~/.config/luban/blueprints/dev.lua    :: edit Lua DSL
luban bp apply dev                        :: applies the blueprint, creates a new generation
luban bp ls                                :: see the stack
```

## Common pitfalls

- **wasm-app without emscripten layer**: `luban new wasm-app` succeeds, but `luban build` fails when `emcmake` isn't on PATH. Fix: `luban bp apply embedded:cpp-base-wasm`.
- **Directory already exists**: `luban new` refuses to write into a pre-populated dir. Either pick a different name or `rm -rf` first.
- **Name collision with vcpkg port**: e.g., `luban new app fmt` — your project's `vcpkg.json` `"name": "fmt"` then conflicts with the fmt port if you `luban add fmt`. Avoid using vcpkg port names as project names.
